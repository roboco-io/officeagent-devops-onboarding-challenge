# OfficeAgent DevOps/AIOps 채용 과제

> 주니어 DevOps/AIOps (정규직). 첫 프로젝트: **OfficeAgent 멀티 환경 배포 설계 — NHN 클라우드·오픈스택 등 고객 규제 대응 환경 지원**.
> 회사의 모든 개발자는 **바이브 코딩 기반 풀스택+풀사이클 개발자**를 지향합니다. 이 채용은 그 첫걸음을 함께할 사람을 찾는 자리입니다.

## 채용 개요

NextIntelligence는 OfficeAgent를 정부기관·공공기관 고객에게 납품하려 합니다. 고객마다 다른 규제 요건(CSAP 등급·망분리·데이터 주권 등)을 충족하려면, OfficeAgent가 **NHN 클라우드, 오픈스택 기반 온프레미스 등 여러 배포 환경을 지원**할 수 있어야 합니다. 현재 AWS에서 운영 중인 OfficeAgent를 이런 규제 대응 환경에도 배포 가능하도록 만드는 것이 이 직무의 첫 프로젝트입니다.

이 프로젝트를 맡으면서, 장기적으로 회사의 다른 모든 개발자처럼 **바이브 코딩 기반 풀스택+풀사이클 개발자**로 성장할 정규직 주니어를 채용합니다.

- **고용 형태**: 정규직 (FTE)
- **시니어리티**: 신입 ~ 1년 (부트캠프 수료자·전환자 환영)
- **멘토링**: 30년+ AWS 경력 시니어 엔지니어가 코칭
- **채용 채널**: 회사가 멘토링하는 인력풀 내 추천·직접 컨택
- **제출 마감**: **2026-05-31 (일) 23:59 KST** (리서치·설계·발표 자료 준비 포함)

## 0. 평가의 핵심

### 메타 목표 (이 과제가 진짜로 묻는 질문)

**"바이브 코딩(AI 페어 프로그래밍)과 AI 기반 학습을 도구로 다뤄, 본인이 익숙하지 않은 영역(NHN·CSAP·오픈스택·정부기관 납품 요건)의 멀티 환경 배포 설계를 실제로 해낼 수 있는 사람인가."**

회사의 모든 개발자는 이 방식으로 일합니다. 이 채용은 그 방식으로 첫 프로젝트를 함께할 사람을 찾는 자리입니다. 따라서 평가자는 모든 산출물에서 **"AI를 어떻게 활용했는가"의 흔적**(retros/ 프롬프트, 의사결정 개정 이력, AI 응답을 비판·수정한 흔적)을 함께 봅니다. **AI 사용을 숨기거나 회피하지 마세요 — 오히려 명시적으로 드러내는 것이 강한 시그널입니다.**

### 3가지 세부 능력

위 메타 목표를 다음 3가지 능력으로 분해해 평가합니다. PRD의 평가 기준 표(§4)와 일대일 매핑됩니다.

| # | 평가 능력 | 핵심 질문 (메타 목표 + 능력별) | §4 채점 영역 매핑 |
|---|----------|--------------------------------|-------------------|
| 1 | **아키텍처 설계능력** | AI와 함께 낯선 제약(NHN·CSAP·오픈스택·정부 납품) 아래서 의사결정·트레이드오프·롤백을 직접 그려낼 수 있는가 | 마이그레이션 설계 30% + 멀티 환경 추상화 20% = **50%** |
| 2 | **구현 및 검증능력** | AI를 사용해 설계를 검증 가능한 작은 artifact (IaC skeleton, 리허설, 무결성 쿼리, runbook 등)로 떨어뜨릴 수 있는가 | 구현 · 검증 가능성 **20%** |
| 3 | **잘 모르는 분야 학습능력** | AI를 학습 가속기로 활용해 1차 자료를 정독하고, 미지 영역을 분해해 본인 설계를 갱신할 수 있는가 | 조사 · 학습 깊이 **20%** (+ Track C 20점 보조) |
| — | (보조) 발표·소통 | 위 3가지를 30분 Q&A에서 — AI에 의존하지 않고 — 본인 머리로 설명할 수 있는가 | 발표 10% |

발표는 별도 능력이 아니라 **위 3축의 검증 수단**입니다. "AI가 만든 문서를 그대로 가져왔는가 vs 본인이 AI와 협업하면서 진짜 이해했는가"를 가장 직접적으로 식별하는 자리.

## 과제 본질

별첨 [`AWS_BASELINE.md`](#부록-별첨-aws-스펙-골격)의 OfficeAgent AWS 구성도를 출발점으로, **NHN 클라우드와 오픈스택 온프레미스 등 고객 규제 환경에도 배포 가능하도록 하는 멀티 환경 설계 + 검증 산출물**을 작성하고 발표 세션에서 의사결정·트레이드오프를 설명하세요.

**주니어 채용입니다.** "정답을 아는가"가 아니라 "**낯선 요건을 어떻게 공부·조사·분해하고, 그 결과를 검증 가능한 형태로 떨어뜨리는가**"가 핵심입니다.

핵심 원칙:
- **검증 가능한 작은 artifact가 필수** — 완성형 IaC나 실 apply는 불필요. 대신 IaC skeleton + `validate` 통과 / 리허설 체크리스트 / 데이터 무결성 쿼리 / runbook 중 **최소 1개 이상**을 본인이 깊이 다룬 도메인 2개에 대해 제출.
- **핵심 결정에는 잠정 결론을 내세요** — 미확인 영역은 "후속 조사 필요"로 넘기지 말고 **리스크 + 검증 계획**을 함께 적습니다.
- **넓고 얕게보다 좁고 깊게** — 5개 도메인을 모두 다루기보다 2개를 정해 깊이 파는 편이 높이 평가됩니다.

## 1. 필수 제출물

### 1.1 `MIGRATION_PLAN.md` — 마이그레이션 설계서

다음 5개 도메인 중 **최소 2개(2개 권장)를 "깊은 분석" 수준**으로 다루세요. 나머지는 개요·매핑·미해결 가정 정도면 됩니다. 욕심내 3개 이상을 얕게 다루기보다 2개를 깊이 파는 편이 높이 평가됩니다.

| 도메인 | 깊은 분석 시 기대 |
|--------|------------------|
| A. 네트워크 / 보안 | VPC ↔ NHN VPC 대응, 망분리 zone, IAM ↔ NHN 권한, CSAP 요건 반영 |
| B. 데이터 / 스토리지 | RDS ↔ NHN DB, S3 ↔ Object Storage, 데이터 이전 단계·다운타임·일관성 검증 |
| C. 컴퓨트 / 배포 | ECS Fargate ↔ NHN K8s/VM, 배포 파이프라인, 블루/그린·롤백 |
| D. 운영·관측 / AIOps | CloudWatch ↔ NHN 모니터링·로그, 알람 진단 자동화, AIOps 통합 |
| E. 비용 / 규제 | 인스턴스·트래픽·라이선스 비용 비교, CSAP·데이터 주권 영향 |

공통 포함:
- AWS(현행) · NHN · 오픈스택 **세 배포 환경의 관계와 공통 설계 큰 그림** (요약 한 페이지)
- 각 단계의 **롤백/단축 경로**
- 본인이 "깊이 다룬 도메인"에서 **잠정 결론** 명시
- **공식 1차 자료 인용** (NHN docs · CSAP 가이드 · 오픈스택 docs URL/캡처)

### 1.2 `ARCHITECTURE.md` — 추상화 설계

같은 OfficeAgent 애플리케이션을 AWS·NHN·오픈스택 등 여러 환경에 배포 가능한 구조를 어떻게 분리할지 **다이어그램 + 짧은 본문**.

- NHN-only / 공통 / 오픈스택-only 영역 구분
- 의사결정 흔적
- 다이어그램 (mermaid / draw.io / Excalidraw / 손그림 스캔 자유)

### 1.3 `VALIDATION.md` (또는 `MIGRATION_PLAN.md` 내 §검증 계획 섹션) — 검증 산출물

**본인이 깊이 다룬 도메인 2개에 대해 각 1개 이상**의 검증 증거를 제출. 형태는 자유 (택 1 이상):

| 증거 유형 | 예시 |
|-----------|------|
| **IaC skeleton** | Terraform/OpenTofu 모듈 + `terraform validate` 또는 `tofu plan` 통과 산출물 |
| **마이그레이션 리허설 체크리스트** | "T+0~T+4h 안에 어떤 명령을 어떤 순서로, 실패 시 어디서 멈추는가" |
| **데이터 무결성 검증 쿼리** | RDS → NHN DB 이전 후 row count·해시 비교 SQL |
| **Object Storage sync dry-run** | `aws s3 sync --dryrun` ↔ NHN CLI 대응, 차이 검증 절차 |
| **알람 진단 pseudo-runbook** | "이 알람이 울리면 어떤 명령으로 어떤 결과를 확인, 어떤 분기" |
| **롤백 리허설 시나리오** | 단계별 트리거 조건 + 명령 + 기대 출력 + 실패 시 판단 기준 |
| **비용 산정 스프레드시트** | NHN/AWS 인스턴스·트래픽·스토리지 비용 정량 비교 |
| **관측·메트릭 검증 정의** | "마이그레이션이 성공했음을 어떤 메트릭이 얼마 동안 유지하면 확정하는가" |

**작성 원칙**: "이 검증을 실제로 실행한다면 **입력·명령·기대 출력·실패 시 판단 기준**이 명확한가"가 발표 Q&A에서 검증됩니다. 짧아도 좋습니다 (도메인당 2~3개 시나리오면 충분).

### 1.4 발표 자료 (`deck/` 또는 단일 PDF)

- **30분 발표 + 30분 Q&A** 가정
- 핵심 의사결정 3~5개를 트레이드오프 표로 요약
- 가장 큰 미해결 위험 1~2개를 솔직히 명시
- 마감 시점에 리포에 포함

### 1.5 `retros/` — 회고 / 학습 일지

`challenge-template/`의 Retrobot이 push 시점마다 KPT 회고를 자동 생성. 본인이 AI에 입력한 프롬프트·세션·시행착오가 추출되어 **학습 궤적의 1순위 사료**가 됩니다.

**Retrobot 또는 유료 AI 구독을 사용하지 않는 경우** 동등한 학습 일지(`retros/*.md`)를 직접 작성하세요. **평가상 불이익 없습니다.**

## 2. 선택 제출물 — §1.3 검증 점수의 근거가 되는 선택 증거

§1.3 검증 산출물의 형태로 다음을 추가 제출할 수 있습니다. **별도 가산점은 없지만, §1.3의 만점 근거가 풍부해집니다.**

- **확장 IaC PoC** (`infra/`): skeleton을 넘어 다중 환경 분기·변수화·실 plan 산출까지
- **운영 자동화 PoC** (`runbooks/`, `scripts/`): 알람 진단·드리프트 비교·비용 이상치 감지에 LLM SDK 통합
- **관측 정의** (`observability/`): 메트릭/로그 대시보드 정의, 알람 룰셋

§1.3이 비어 있는데 §2만 화려한 산출물은 평가 무게가 약합니다 — 검증의 본질은 양이 아니라 "이 계획이 맞는지 어떻게 증명할 것인가"입니다.

## 3. 기술 조건

| 항목 | 조건 |
|------|------|
| IaC 도구 | **자유** (Terraform / OpenTofu / Pulumi / Crossplane / Ansible 등). 사용 시 NHN+오픈스택 추상화 사고가 입증되어야 함 |
| 클라우드 계정 | **불필요** — 실 apply 없음. `validate` / `plan` / `dry-run` 등 **로컬 검증 산출물**로 충분 |
| AI 에이전트 | **Claude Code 권장**. Codex CLI·Gemini CLI 등 다른 에이전트도 사용 가능 — 규칙·스킬·훅 컨버팅 안내는 템플릿 `docs/AGENT_SETUP.md` 참조. 어떤 에이전트를 쓰든 평가 불이익 없음 |
| 회고 도구 | Retrobot 권장 (claude→codex→gemini 자동 폴백). **유료 AI 구독이 없으면** 수동 `retros/*.md` 작성으로 대체 — 평가 불이익 없음 |
| 제출 방식 | GitHub 비공개 리포에 `serithemage` collaborator 초대 (부득이하면 zip). 발표 자료 포함 |
| 발표 세션 | 제출 마감 후 7일 이내 화상회의 30분 발표 + 30분 Q&A |

> **에이전트 선택 안내**: 템플릿의 규칙(`CLAUDE.md`)·스킬(`retrobot/SKILL.md`)·훅(`.githooks/post-commit`)은 Claude Code 기준으로 작성되어 있고, `AGENTS.md`·`GEMINI.md` 심볼릭 링크와 훅의 다중 에이전트 폴백으로 Codex·Gemini도 바로 동작합니다. 본인 에이전트에 맞춰 컨버팅한 흔적(예: 훅 분기 추가)을 retros에 남기면 §4 조사·학습 / Track C에서 긍정 평가 요소가 됩니다.

## 4. 평가 기준

| 영역 | 비중 | 평가 능력 | 주요 관점 |
|------|-----:|----------|----------|
| **마이그레이션 설계 · 의사결정** | **30%** | 아키텍처 설계 | 깊이 다룬 2개 도메인의 잠정 결론, 단계·롤백·다운타임 전략, 트레이드오프 |
| **멀티 환경 추상화** | **20%** | 아키텍처 설계 | NHN-only / 공통 / 오픈스택-only 분리, 장기 전환 시 재작성 지점 식별 |
| **구현 · 검증 가능성** | **20%** | 구현·검증 | §1.3 검증 산출물의 재현성·구체성 (입력·명령·기대 출력·실패 판단 기준) |
| **조사 · 학습 깊이** | **20%** | 잘 모르는 분야 학습 | 1차 자료 인용 충실도, 미지 영역 분해, **학습 후 설계 변경 흔적**, 대안 비교 |
| **발표 · 소통** | **10%** | 보조 (위 3축 검증 수단) | 자료 구조, 30분 Q&A에서 의사결정·검증 근거 설명, 미해결 위험 솔직성 |

학습 궤적·바이브 코딩 흔적(retros/)은 **Track C (별도 20점, 우선순위 조정 지표)**로 산출합니다. 60~75점 경계 후보의 합격·조건부 판단에 결정 카드로 사용. 채점 세부는 회사 내부 평가 가이드에서 관리되며, 응시자는 본 §4 평가 비중과 [`TEMPLATE_GUIDE.md`](./TEMPLATE_GUIDE.md)를 기준으로 작성하시면 됩니다.

**완성도보다 의사결정·조사·검증 흔적이 우선**. 단, **핵심 결정**에는 잠정 결론과 검증 계획이 필요합니다 — "후속 조사 필요"만 적은 채 미답으로 두면 §불합격 기준 후보입니다.

## 7. 자주 묻는 질문

**Q. AWS 인프라 스펙은 어디서 받나요?**
별첨 §부록 "별첨 AWS 스펙 골격" 또는 공고 시 같이 제공되는 익명화 버전을 출발점으로 합니다.

**Q. NHN 계정이 필요한가요?**
**불필요**. 실 apply 없음. 설계 + `validate`/`plan`/`dry-run` 등 로컬 검증 산출물로 충분.

**Q. IaC 도구를 모르는데 시작해도 되나요?**
가능합니다. **마감일까지 어떻게 학습·조사하는지** 자체가 평가 대상입니다. retros에 학습 궤적을 남기세요. 코드 형태의 IaC가 부담스러우면 §1.3에 **체크리스트·리허설·검증 쿼리** 같은 문서형 증거로 대체 가능합니다.

**Q. AI 코딩 도구가 필수인가요?**
권장이지만 **필수 아님**. 유료 구독이 없으면 수동 `retros/*.md`로 대체 가능, 평가 불이익 없음.

**Q. 코드 구현은 꼭 해야 하나요?**
**완성형 IaC는 필수가 아닙니다.** 단, **§1.3 검증 산출물**은 필수입니다 — 코드형(IaC skeleton, plan/dry-run)도 좋고, 문서형(리허설 체크리스트, 검증 쿼리, runbook)도 동등하게 평가됩니다.

**Q. 5개 도메인을 다 다뤄야 하나요?**
아니오. 2개를 정해 깊은 분석 + 검증 산출물을 붙이고, 나머지는 매핑·미해결 가정 수준이면 충분합니다.

**Q. "모른다"고 적으면 감점인가요?**
미확인 자체는 감점이 아닙니다. 단, **핵심 결정**(깊이 다룬 도메인)에는 잠정 결론이 필요하고, 미확인이라면 리스크 + 검증 계획을 함께 적어야 합니다. 단순 미답은 감점.

**Q. 발표 자료 형식은?**
PDF·Keynote·Google Slides·Markdown deck 자유. 마감 시점에 리포에 포함되어 있어야 합니다.

## 부록: 별첨 AWS 스펙 골격

OfficeAgent의 운영 인프라(익명화 버전)는 다음 수준입니다. 후보는 이 골격을 출발점으로 마이그레이션 설계를 진행합니다. **실 운영 인프라 전체가 아니라 후보 평가 범위로 한정된 가상 스펙입니다.**

```
[AWS 리전: ap-northeast-2]
│
├── Network
│   ├── VPC (10.0.0.0/16)
│   ├── Public subnet × 2 (ALB·NAT Gateway)
│   ├── Private subnet × 2 (앱·DB)
│   └── Security Groups (ALB → App → DB 흐름 제한)
│
├── Compute
│   ├── Application Load Balancer (ALB)
│   ├── ECS Fargate Service × 1 (FastAPI 백엔드, 2 task)
│   └── ECR (컨테이너 레지스트리)
│
├── Data
│   ├── RDS PostgreSQL (Multi-AZ, db.t3.medium)
│   ├── S3: 문서 업로드 버킷 + 정적 자산 버킷
│   └── ElastiCache Redis (캐시·세션, single node)
│
├── Identity & Secrets
│   ├── IAM Role (ECS Task Role / Execution Role)
│   └── AWS Secrets Manager (DB credential, API key)
│
├── Observability
│   ├── CloudWatch Logs (앱·ALB 액세스)
│   ├── CloudWatch Metrics + Alarms (CPU·error rate)
│   └── X-Ray (선택, 일부 트레이스)
│
├── CI/CD
│   ├── GitHub Actions (build + ECR push + ECS deploy)
│   └── Trivy 이미지 스캔
│
└── External
    ├── Anthropic API · OpenAI API (LLM 호출)
    └── 외부 SaaS (Slack notification)
```

**가상 트래픽 규모**: 동시 사용자 < 200, 일일 RPS 평균 5 / 피크 50, 문서 업로드 일평균 500건.

**가상 컴플라이언스 요건**: 향후 공공 부문 도입을 위해 CSAP 인증 환경 대응 필요. 망분리·암호화·감사 로그 요건이 NHN 단계부터 적용된다고 가정.

**가상 운영 제약**: 마이그레이션 동안 다운타임 허용 시간 **최대 4시간** (주말 야간 1회). 데이터 이전 무손실 필수.

## 부록 B: 공공·정부기관 납품 규제 — LLM Wiki 안내

> **본 채용의 핵심 동기**: OfficeAgent는 정부기관·공공기관 납품을 목표로 합니다. 고객마다 다른 규제 요건(CSAP 등급·망분리·데이터 주권)을 충족하려면 NHN 클라우드, 오픈스택 온프레미스 등 여러 배포 환경을 지원해야 하며, 이것이 본 직무가 풀어야 할 문제입니다.

공공 납품요건의 **상세 지식 베이스는 과제 템플릿 리포의 `wiki/` 디렉토리에 LLM Wiki로 구조화**되어 있습니다. `wiki/index.md`가 진입점이며, 각 페이지는 `[[wiki-link]]` 교차참조와 1차 자료 출처를 갖춥니다.

### B.1 wiki 활용 방법

- 응시자·AI 에이전트는 `wiki/index.md`를 진입점으로, 관련 페이지를 따라 읽으며 규제 맥락을 파악합니다 (`CLAUDE.md` §"LLM Wiki 활용 규칙" 참조).
- wiki는 **출발점**입니다. 평가의 핵심(§4 조사·학습 깊이)은 wiki를 기반으로 1차 자료까지 직접 파고드는 능력입니다. wiki를 그대로 베끼는 것은 평가되지 않습니다.
- 조사하며 새로 알게 된 내용을 `llm-wiki ingest`로 wiki에 추가하면 §4 조사·학습 / Track C에서 긍정 평가 (필수 아님).
